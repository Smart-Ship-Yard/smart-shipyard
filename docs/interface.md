# 젯슨 → 서버 통신 스펙 (v1.2)

젯슨(UGV)이 백엔드 서버로 보내야 하는 데이터 전부와 그 형식.
연산(추론·좌표 융합·판별)은 전부 젯슨에서 끝내고, 서버는 결과를 중계·저장만 한다.

- **보내는 것 5종** (젯슨 → 서버): ① 위치 핑 · ② 위험 이벤트 · ③ 조립 단계 · ④ 배 위치 — JSON / ⑤ 실시간 영상 — 바이너리
- **받는 것 1종** (서버 → 젯슨): ⑥ `stream_boost` — 영상 화질 올려/내려 명령

## 접속 주소

- JSON 채널: `ws://<서버IP>:8000/ws/jetson` — ①~④ 전송 + ⑥ 수신
- 영상 채널: `ws://<서버IP>:8000/ws/jetson-stream` — ⑤ 전송 (바이너리 전용, JSON과 분리)
- (참고) 프론트엔드용: `ws://<서버IP>:8000/ws/frontend`(JSON), `ws://<서버IP>:8000/ws/frontend-stream`(영상)
- 서버 IP는 같은 공유기 내부 IP라 그때그때 공유 (공유기에서 DHCP 고정 할당 예정)

## ① 위치 핑 — DB 저장 안 함

평상시 계속 전송. 주기는 일단 0.5초에 1번으로 시작 (조절 가능).

```json
{"event_type": "position", "ekf_global": [3.2, 7.8]}
```

- `ekf_global` = map 기준 차체 절대좌표 `[x, y]` 미터 — ekf_global(융합) 출력값

## ② 위험 이벤트 — DB 저장 ✅

감지 시에만 전송. conf 0.5 미만은 버림. 같은 상황을 프레임마다 연사하지 말고 안정 감지 시 1번.

```json
{
  "event_type": "fallen_person",
  "confidence": 0.91,
  "depth_xyz": [1.2, 0.4, 2.1],
  "ekf_global": [3.2, 7.8]
}
```

- `event_type` 허용 값 4개: `fallen_person` · `fire` · `no_helmet` · `ship_defect` (모델 클래스 이름 그대로)
- `person`, `helmet` 같은 정상 상태 클래스는 **보내지 않음**
- `confidence` = YOLO conf 그대로 (0.5~1.0)
- `depth_xyz` = 카메라 기준 객체 3D 좌표 `[X, Y, Z]` 미터 — depth_yolo_test.py에서 계산하는 그 값, 배열 하나로
- `ekf_global` = 감지한 순간의 차체 위치
- `timestamp`는 서버가 자동으로 붙임 → **보내지 마**

## ③ 조립 단계 변화 — DB 저장 ✅

단계가 바뀌었을 때 1번 + 부팅·재접속 시 현재 단계 1번.

```json
{"event_type": "block_level", "block_id": "B1", "level": 2}
```

- `level` = 클래스 이름(level1/2/3)이 아니라 **숫자만** 1, 2, 3
- `block_id` = 당분간 `"B1"` 고정 (배 1척이라)
- YOLO가 프레임마다 흔들리니, 몇 초간 같은 단계로 안정 감지됐을 때만 "바뀜"으로 인정

## ④ 배 위치 (측량 결과) — DB 저장 ✅

세션 시작 측량 후 1번 + **조립 단계가 바뀔 때마다** 재측량 후 1번 (직접 조립하는 과정에서 배가 밀리거나 돌아갈 수 있음).

```json
{"event_type": "ship_pose", "block_id": "B1", "map_xy": [5.1, 4.8], "yaw": 1.57}
```

- `map_xy` = 배 중심의 map 좌표 `[x, y]`, 미터
- `yaw` = 배 방향각, 라디안 (map x축 기준 반시계)
- 실시간 아님 — 측량(맵에서 배 지정 등)이 끝났을 때만 1번씩
- 서버가 저장해뒀다가 프론트 초기 로딩(init-data)에 제공 → 3D 배 모델 배치에 쓰임

## ⑤ 실시간 영상 — DB 저장 안 함

상시 전송. YOLO 추론에 쓰는 프레임을 재활용 (카메라를 따로 또 열지 않음).

- JPEG 인코딩 후 **바이너리 프레임**으로 영상 채널에 전송
- **평상시**: 480×360 · JPEG 품질 60 · 초당 5장 (~1 Mbps)
- **부스트 모드**: 640×480 · 품질 80 · 초당 15장 (~6 Mbps) — 아래 ⑥ 명령을 받았을 때만

## ⑥ stream_boost 명령 (서버 → 젯슨)

프론트가 영상 팝업을 크게 열면(위험 이벤트 시 자동 + 사용자가 미리보기 클릭) 서버가 전달함.

```json
{"event_type": "stream_boost", "action": "start"}
```

- `action: "start"` → 부스트 모드로 전환
- `action: "stop"` → 평상시 화질로 원복 (프론트에서 닫기 버튼 눌렀을 때 옴)
- 안전장치: start 후 **5분** 동안 stop이 안 오면 스스로 원복 (프론트 비정상 종료 대비)

## JSON 키 전체 요약

| 키 | 타입 / 허용 값 | 들어가는 메시지 |
|---|---|---|
| `event_type` | 문자열 — 전송: position · fallen_person · fire · no_helmet · ship_defect · block_level · ship_pose / 수신: stream_boost | 모든 메시지 (필수) |
| `ekf_global` | 숫자 2개 배열 [x, y], 미터 | ① 위치 핑, ② 위험 이벤트 |
| `confidence` | 숫자 0.5~1.0 | ② 위험 이벤트 |
| `depth_xyz` | 숫자 3개 배열 [X, Y, Z], 미터 | ② 위험 이벤트 |
| `block_id` | 문자열, `"B1"` 고정 | ③ 조립 단계, ④ 배 위치 |
| `level` | 정수 1 · 2 · 3 | ③ 조립 단계 |
| `map_xy` | 숫자 2개 배열 [x, y], 미터 — 배 중심 | ④ 배 위치 |
| `yaw` | 숫자, 라디안 — 배 방향각 | ④ 배 위치 |
| `action` | `"start"` · `"stop"` | ⑥ stream_boost (수신) |
| `timestamp` | 서버가 자동 추가 — 젯슨은 생략 | DB 저장 대상 6종 |

## 공통 규칙

- **철자 정확히.** 서버는 모르는 event_type을 에러 없이 조용히 무시함 — 오타 나면 아무 일도 안 일어나서 디버깅 지옥임. 위 표에서 복붙 권장.
- 사진/스냅샷을 이벤트 JSON에 넣지 않음 — 영상 채널이 항상 흐르니까 프론트가 거기서 봄.
- JSON은 텍스트 프레임, 영상은 바이너리 프레임 — 채널을 섞지 말 것.

## 좌표계 메모

- `ekf_global`의 원점·축 정의(맵 만드는 방식)는 슬램 쪽에서 확정한 그대로 가되, 프론트(3D 매핑)와 **직접** 공유 필요. 서버는 좌표를 변환하지 않고 그대로 중계·저장만 함.
- 배(블록) 위치는 ④ `ship_pose` 메시지로 정식 편입됨 — 세션 시작 + 조립 단계 변경 시마다 재측량해서 전송. 프론트는 이 값으로 3D 배 모델을 배치하므로, 이게 틀어지면 로봇 마커가 배를 뚫고 다니는 화면이 됨.
